# 项目开发原则

本原则适用于后续所有人工、Codex 与 AI 辅助开发任务。

## 1. 单一职责

- 页面不同时承担数据库、业务计算和 UI 渲染。
- 一个模块只承担一个主要职责，文件名和导出接口应表达该职责。
- 避免万能组件、万能 service 和不断增长的条件分支。

## 2. 分层与依赖方向

推荐边界为：

~~~text
UI
  -> application/service

application/service
  -> domain/pure transformation
  -> persistence adapter
      -> Dexie / external system
~~~

- domain/pure transformation 不依赖 UI、React、Zustand、Dexie 或浏览器持久化 API。
- persistence adapter 不依赖 UI。
- application/service 负责编排纯业务规则和持久化接口。
- UI 不直接访问 Dexie。
- 禁止纯函数读取全局状态、service 操控 React，以及布局算法混入数据查询。

## 3. AI 辅助开发约束

- AI 生成代码必须经过项目边界检查，发现已有能力优先复用。
- 不因生成方便复制实现、随手塞进最近文件或用大段条件分支代替模块设计。
- 不以测试通过接受明显冗余；不得未经确认创建平行 service、store 或 utils。
- 每项任务说明新增依赖、耦合点和删除影响；完成后检查命名、重复、死代码和未使用导出。

## 4. 抽象原则

- 出现真实重复后再抽取公共模块，不为假设的未来建立复杂框架。
- 只有一个调用者且没有清晰边界价值时，不建立抽象。
- 接口只隔离容易变化的实现，例如布局算法与外部 API；抽象必须提升可测试性、可替换性或可删除性。

## 5. 可删除性

每个功能必须说明入口、数据所有权、依赖与被依赖模块、删除文件范围、删除后残留的字段/设置/任务/依赖，以及是否能独立移除。可选功能不得仅为 UI 便利写入核心表，除非有明确业务必要。

## 6. 数据库原则

- Dexie schema 变化必须有业务理由和迁移测试。
- 派生数据优先运行时计算，核心事实与展示缓存分离。
- 删除表或字段前先分析备份、恢复和迁移兼容性。

## 7. 状态管理原则

- 局部状态优先于全局状态；多个组件存在不等于需要 Zustand store。
- 持久化状态必须说明用户价值；临时 UI 状态不得写数据库。
- 过期异步请求不得覆盖当前状态。

## 8. 测试原则

- 纯业务规则用纯函数测试，service 测试数据库边界，组件测试交互和状态，E2E 只验证关键路径。
- 不用 E2E 替代快速单元测试；不删除合理断言来让 CI 变绿。
- 修复生产代码而非降低测试标准；删除功能时同步删除或更新测试。

## 9. 错误处理原则

- 错误限制在最小功能边界，子功能失败不应让整个应用不可用。
- 错误信息必须可理解，可恢复错误提供重试。
- 防御异常数据，但不静默吞掉影响用户结果的错误，也不擅自修改用户数据。

## 10. 外部依赖原则

引入依赖前检查当前代码是否确有必要、包大小、维护活跃度、许可证、API 稳定性、安全风险、删除成本，以及是否仅为小功能引入大型框架。每个新增依赖必须在完成报告中说明。

## 11. 参考优秀项目和源码

可参考官方文档、官方示例、原始项目源码、维护活跃且测试充分的开源项目。先说明当前问题和参考方案解决的边界，只参考思想而不机械复制不适合当前规模的架构；不得隐藏来源或不合规复制许可证代码。架构取舍写入设计文档，优先使用一手来源。

## 12. 修改范围原则

- 只修改完成任务必需的文件，禁止无关重构、顺手大规模重命名和混合独立功能。
- 一个提交表达一个清晰意图；大功能先设计、再计划、后实现。

## 13. 文档原则

- 架构变化更新设计文档，用户可见行为更新 README，数据格式变化更新备份说明。
- 文档不得包含失效命令、未完成的占位文字或模糊占位。

## 14. 完成前检查

完成前检查是否重复实现已有能力、引入不必要依赖、扩大全局状态、跨越层边界、留下死代码/未使用导出/调试日志；确认可独立删除、核心规则有测试、文档同步，并确保 git diff 只包含任务范围。

## 15. 删除功能检查表

删除功能前列出并评估：路由、页面与组件、service、store、数据库表与索引、设置项、API、定时任务、测试、文档、npm 依赖、备份格式、迁移兼容性、残留导入和未使用代码。
