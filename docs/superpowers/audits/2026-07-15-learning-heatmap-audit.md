# Task 20-B0 学习热力图现状、数据口径与视觉扩展审计

审计基线：`f3fd21c fix: coordinate editor saves with ai application`。本报告只基于当前代码、schema 和测试读取结果；“目标设计”中的学习时长、练习、复盘、旧知识复习、课程筛选和学习会话均不被当作现有能力。

## 1. 总体评价

当前热力图是一个可用的“笔记创建足迹”组件，而不是完整的学习活动热力图。它没有独立活动表、没有时长或复盘数据，也不会统计笔记编辑、AI 或知识模型变更。

- **可用性：B-**。独立入口、首页摘要、日期点击和原生按钮可用。
- **数据诚实度：C**。实现实际计数的是活动笔记的创建数，但“学习热力图 / 学习足迹”的文案容易被理解为学习时长或每日学习活动；总笔记数与活跃天数也没有限制为当前展示窗口。
- **扩展准备度：C**。当前组件小、可删除、无第三方耦合，但没有纯日期聚合边界、筛选契约或活动数据模型。
- **性能：当前规模低风险**。完整视图 182 个格子、首页 140 个格子；主要成本来自应用启动时和笔记变更后对 `notes` 全表的读取/聚合，而不是格子渲染。

## 2. 当前入口与组件架构

真实入口有两个：

```text
Sidebar “🔥 学习热力图” ──> /heatmap
HomePage “查看学习足迹” ────> /heatmap
HomePage 仪表卡片 ───────────> Heatmap(compact)

App
  ├─ lazy(HeatmapPage) for /heatmap
  └─ App mount: noteStore.loadAllNotes()
       └─ noteService.fetchNotes()
            └─ db.notes.toArray()

HeatmapPage
  └─ Heatmap(onSelectDate)
       └─ useNoteStore(s => s.allNotes)
            └─ dateMap / weeks / monthLabels
                 └─ DOM button grid
                      └─ navigate('/?date=YYYY-MM-DD')
                           └─ HomePage -> noteService.fetchNotes({ createdDate })
```

- `src/pages/HeatmapPage.tsx` 是独立路由页，标题为“学习热力图”，只有返回首页、说明文字和一个 `Heatmap`。
- `src/pages/HomePage.tsx` 在无筛选且 `allNotes.length > 0` 时内嵌 `Heatmap compact`，它是辅助 Dashboard 卡片而非首页主内容。
- 热力图未与 AI 历史、知识图谱或实体详情同页混合。
- Sidebar 提供独立导航；Home Hero 也提供入口。
- 没有日期详情页面或抽屉：点击格子只跳转首页的 `?date=` 笔记创建日期筛选。
- 页面没有年份/月度选择、指标选择、课程筛选、两年比较、加载骨架或独立错误状态。

## 3. 当前数据来源

### 已实际使用的数据

|链路环节|当前实现|结论|
|---|---|---|
|原始表|`db.notes`，即活动笔记表|唯一来源|
|装载|`App` 调用 `noteStore.loadAllNotes()`|应用启动时一次全量读取|
|查询|`noteService.fetchNotes()` 无 filter 时 `db.notes.toArray()`|未按日期范围读取|
|聚合|`Heatmap` 遍历 `allNotes`，以 `note.createdAt.split('T')[0]` 分组|每篇活动笔记最多贡献一次|
|强度|日创建数映射为 0–4 级|固定阈值|
|点击查询|`HomePage` 传入 `{ createdDate }`；`noteService` 对已读 notes 再 `slice(0, 10)` 过滤|仍是全表后内存过滤|

### 当前不计入的数据

|数据或事件|当前是否计入|代码证据|
|---|---|---|
|笔记 `updatedAt`、一次或多次编辑|否|`Heatmap` 只读取 `createdAt`|
|回收站 `deletedNotes`|否|组件只读 `allNotes`；该状态只由活动 `notes` 装载|
|已删除后恢复的笔记|恢复后按原始 `createdAt` 计一次|`restoreDeletedNote()` 只更新 `updatedAt`；热力图不读它|
|AIResult 创建、应用、放弃、过期|否|未读取 `aiResults`|
|KnowledgeAuditLog|否|未读取 `knowledgeAuditLogs`|
|实体、关系、笔记—实体关联变化|否|未读取相关表|
|视频播放进度|否|不属于热力图链路|
|学习时长、会话、练习、复盘、掌握度|当前不存在对应持久化模型|schema/type 中没有相关表或字段|

同一天创建并多次编辑同一篇笔记，值仍为 1；同一天创建多篇笔记，按篇数累加。创建与修改不会重复计算。删除活动笔记后该格子的贡献会消失；恢复后回到原创建日期，不会在恢复日期重复累计。

`createdAt` 虽已在 Dexie `notes` schema 中建索引，但当前链路没有使用该索引限定日期范围。热力图本身不再次访问 Dexie，却依赖全局 `allNotes` 快照；笔记新增、删除或 `loadAllNotes()` 更新时会重新聚合。

## 4. 当前统计口径

**每天的真实含义：当天（UTC 日期）创建且当前仍在活动笔记表中的笔记数量。** 两种 `NoteType`（`knowledge_fragment`、`course_chapter`）均计入，不区分课程、项目、目录或标签。

固定强度函数在 `Heatmap.tsx` 内部：

|日创建数|level|颜色|
|---:|---:|---|
|0|0|`var(--surface)`|
|1|1|20% accent 蓝|
|2–3|2|40% accent 蓝|
|4–5|3|70% accent 蓝|
|≥6|4|`var(--accent)`|

它是固定分桶而非个人基线或动态归一化，因此不同月份、不同年份可按“创建篇数”比较；但极端日会压缩到 level 4，无法区分 6 篇与 60 篇。零活动和没有数据不区分，都会显示 level 0。

三个按现有算法推导的例子：

1. `2026-07-15T02:00:00.000Z` 创建一篇活动笔记，`dateMap['2026-07-15'] = 1`，格子为 level 1；当日之后即使编辑十次也不改变它。
2. 两篇不同活动笔记均在 `2026-07-16` 创建，值为 2，格子为 level 2；若其中一篇移入回收站，重载后的值为 1。
3. 六篇活动笔记在 `2026-07-17` 创建，值为 6，格子为 level 4；计数继续增加也不会产生更深颜色。

`totalNotes = notes.length` 与 `activeDays = dateMap.size` 都是全库累计值，不是可见的最近 20/26 周范围。这与“最近 N 周”副标题并列展示，属于统计范围不一致的误导风险。

## 5. 日期聚合与时间边界

### 当前范围和布局

- 完整页 `weekCount = 26`，渲染 `26 × 7 = 182` 天；首页 compact 为 20 周，即 140 天。当前不是全年、也不是 365 天。
- 日期数组从“组件渲染时的今天减 181 天”到今天，按连续 7 天分列；并没有对齐自然周。
- 左侧却固定显示“一、三、五、日”，逻辑上假定首行是周一、末行是周日。由于起点只是“今天减 181 天”，除非恰好满足该对齐条件，标签会指向错误星期。
- 月份标签读取每列第一个日期，使用 `new Date(week[0]).getMonth()`；同一列跨月时只显示列首月份。
- 没有年份标题、年份切换、月度放大、两年对比、日期选中样式、今天突出样式或未来日期格。

### 时区、DST 与跨年

日期键同时来自 ISO 字符串：笔记创建使用 `new Date().toISOString()`，格子使用 `date.toISOString().split('T')[0]`。这使点击筛选与聚合在 UTC 键上相互一致，却不是本地学习日：例如东八区用户在本地 `2026-07-15 00:30` 创建笔记，ISO 键为 `2026-07-14`，界面会显示前一天。

`new Date('YYYY-MM-DD')` 按 UTC 午夜解析，再用本地 `getMonth()` 取月份；在 UTC 负时区会把月初标签显示为上个月。DST 不会改变 UTC 键本身，但“本地学习日”与 UTC 日的偏差以及 `setDate` 跨 DST 的本地时间移动都没有专门测试。12 月至 1 月与闰日会由 JavaScript `Date#setDate` 自然滚动，但当前没有覆盖它们的测试。

组件只有 `useMemo([weekCount])` 生成 `weeks`，打开页面跨过午夜时不会自行滚动到新的一天，直到组件重新挂载或发生重渲染；也没有定时器，因此不存在后台持续计算，但存在日界面陈旧风险。

## 6. 渲染技术与组件成本

实现是普通 React DOM + inline CSS Grid，不使用 SVG、Canvas、第三方 contribution calendar 或可视化库。

- 完整页创建约 182 个按钮、26 个周列容器、月份/星期标签和 5 个图例色块；首页约 140 个按钮。每格有独立的 `title`、`aria-label` 和 `onClick` 回调。
- 没有独立 Tooltip 组件、ResizeObserver、scroll/resize listener、动画、requestAnimationFrame 或 interval；不存在监听器清理问题。
- `dateMap` 由 `useMemo([notes])` 缓存；`weeks` 由 `useMemo([weekCount])` 缓存；`monthLabels` 依赖 `weeks`。`getLevel` 和颜色数组会在 render 重建，但对 182 格没有实质成本。
- 没有 `React.memo`。任意 `allNotes` 引用变化会让整个热力图重渲染并重建按钮数组；在当前单年以下的 DOM 数量上是低风险，不能据此引入虚拟化。
- 主要可见成本是 `App` 的 `loadAllNotes()` 和笔记变化后的全量数组；在 2,000 篇笔记或多年份活动数据目标下，需要先测量再考虑日期范围查询或预聚合。
- `HeatmapPage` 本身经 `React.lazy` 进入独立路由；但 `HomePage` 静态导入 `Heatmap`，因此首页仍会加载组件代码（组件很小，无第三方库）。

## 7. 当前视觉与信息层级

当前视觉是轻量、统一的蓝色足迹卡片：完整页包在 `surface-card`，首页包在 `dashboard-card--heatmap`；格子圆角 3px/4px、间距 4px，月份/星期标签 10px，图例只有“少—多”和五个色块。色阶为一个 accent 蓝系，和全局 `--surface`/`--accent` 主题变量一致，深色模式也复用变量。

优点：单一强度色系与现有视觉系统相容；完整页与首页同组件，信息层次简单；今日以外的日期不会被误写成未来活动。

局限：

- 全宽 `repeat(N, minmax(0, 1fr))` 在窄窗口会把格子压得过小；没有最小格宽或横向滚动容器。月份文字 `white-space: nowrap` 可能挤压或重叠。
- 宽屏时格子会随容器拉大，缺少专业年度日历所需的稳定密度与年份视觉锚点。
- 没有空状态、加载骨架和错误状态；空库与尚未装载时均是零色方格与 0 统计。
- 格子 `title` 是浏览器原生提示，不是带课程/活动摘要的产品 Tooltip；没有当前日期、选中日期或详情层级。
- 视觉仍接近通用 contribution calendar，尚未形成“学习而非提交记录”的专业语义。

### 与目标设计的对照

|目标项|状态|依据|
|---|---|---|
|单一强度渐变、0–4 级|已具备|已有 5 个固定色阶|
|单年默认全年精致方格|需要局部调整|仅 20/26 周，非全年|
|中性底色|可复用|`--surface` 为 level 0|
|课程颜色只作辅助|当前缺少数据支持|不读取课程归属|
|年份切换、可选两年比较|需要新增能力|没有状态与日期范围契约|
|月度日历卡片|需要新增能力|无月度视图|
|日期摘要与详情抽屉|需要新增能力|只跳首页创建日期筛选|
|指标/课程筛选|当前缺少数据支持|只有创建数；无学习活动类型|
|学习时长、练习、复盘、旧知识复习|当前缺少数据支持|无对应记录|

## 8. 当前交互与无障碍

已具备：每个格子是原生 `button`，拥有 `aria-label` 和原生键盘焦点；全局 CSS 提供 `button:focus-visible` 轮廓；`title` 会显示“日期: N 篇笔记”；点击可以进入首页日期筛选。全局 `prefers-reduced-motion` 规则存在，热力图自身没有动画。

缺失或部分支持：

- Enter/Space 会触发 button 的原生 click，但没有日期详情、选中状态或回到热力图的键盘语义。
- 色彩不是唯一信息载体：图例和 `aria-label` 有数量，但视觉图例没有阈值文本，鼠标用户仍只能从颜色推测等级。
- 页面没有 `role="grid"`/行列语义，也没有明确“当前格是星期几”的可访问名称；错误的星期标签会同时误导视觉用户。
- 原生 `title` 不保证触屏、键盘辅助提示或屏幕阅读器可用；不存在可聚焦/可读的详情内容。
- 无选中态、空态或加载态说明；深色主题色阶是否满足每一级相邻对比度没有测试或测量证据。

综合等级为 **部分支持**：基础按钮和焦点良好，但产品级日期语义、详情、图例说明和小屏可用性基本缺失。

## 9. 与目标设计的差距

目标中的“综合学习强度”不能由当前创建笔记数诚实替代。课程/章节字段虽存在于 `Note`，但热力图没有读取它们；`Course` 只有名称、来源、总章节、视频地址和目录，不含学习活动、练习、课时完成或时长。当前可以诚实地做“笔记创建足迹”的年度版，却不应提前绘制时长、课程贡献、复盘或掌握度 UI。

特别需要先纠正的产品契约：标题/说明应清楚表达“创建笔记”还是将来真实活动；若继续以“学习热力图”呈现，用户会合理期待编辑、视频和复盘也产生足迹。

## 10. 数据模型差距

|目标能力|现状判定|原因|
|---|---|---|
|笔记创建足迹|现有数据可直接支持|活动 `Note.createdAt`|
|笔记编辑足迹|可有限推导但不可靠|只有最后 `updatedAt`，无编辑事件历史|
|课程归属的创建笔记过滤|可从现有数据有限推导|`Note.courseId` 仅对学习单元有意义；自由笔记不一定归属课程|
|学习时长|必须新增持久化模型|没有会话或时间片段|
|练习完成|必须新增持久化模型|没有练习实体/完成事件|
|复盘次数、旧知识复习|必须新增持久化模型|没有复盘事实记录|
|章节/课时|需要进一步产品定义|当前只有 `course_chapter` 笔记和可选 `chapterOrder`，不是完整层级|
|学习会话与活动时间片段|必须新增持久化模型|无开始、结束、暂停或归属|
|掌握度|必须新增持久化模型|现有实体状态是知识审核状态，不是掌握度|
|日期详情|笔记创建部分可直接支持；完整详情需要新增模型|当前只能列出创建笔记|

AIResult 与 KnowledgeAuditLog 拥有时间戳，但它们分别表示 AI 生命周期与知识结构审计，不能被偷换成学习时长或复盘事件。将其混入“综合学习强度”会造成不可解释统计。

## 13. 三种实施路线比较

|方案|价值与边界|成本/真实性|数据库与 Backup|性能/8GB/可撤销性|
|---|---|---|---|---|
|A. 增强当前组件|明确改名为笔记创建足迹，修正自然周/本地日期/统计范围，改善图例、空态、键盘说明与年份浏览|低成本；只反映创建笔记，不能表示真实学习量|无需改 schema 或 Backup|低成本；组件仍可独立删除|
|B. 以当前口径重建专业年度组件|保留“活动笔记创建”唯一数据源，做全年视图、稳定月份、日期摘要、响应式和无障碍；不做时长/课程筛选|中成本；视觉专业但数据仍受限，必须诚实命名|可不改 schema/Backup|一年约 365 格低风险；保留单年默认|
|C. 先建设学习活动与课程数据模型，再实现完整目标|真实支持时长、练习、复盘、旧知识、课程筛选、详情及两年对比|高成本；数据真实性最高，依赖学习计时与复盘产品定义|需要 Dexie 升级、迁移策略和很可能的 Backup v6|需设计索引、范围查询、聚合缓存边界；8GB 设备先测量|

## 14. 推荐阶段路线

推荐 **A → B → C**，而非一次性重做。

1. 先执行低风险的口径修正：把当前数据诚实命名为“笔记创建足迹”，将日期生成和分级提为纯函数并以本地日历作为唯一边界，修正统计范围和星期对齐。
2. 再在不改 schema 的前提下执行年度组件重建：只用活动笔记创建数据，提供单年视图、日期创建笔记摘要、可访问图例和窄屏策略。此阶段不展示不存在的分钟数、练习或课程贡献。
3. 学习计时器、复盘与课程活动模型完成专项设计、持久化和 Backup 评估后，再建设真实综合强度、课程筛选、月度详情与两年比较。

## 15. P0/P1/P2 风险

### P0

### P1

### P2

- 没有空/加载/错误状态、年份/月度/详情、筛选、选中态与响应式最小尺寸。
- 没有组件/日期/无障碍/生产 E2E 覆盖。
- 全量活动笔记读取和主线程聚合在大规模时需要测量，但当前 182 格没有虚拟化理由。
- `new Date('YYYY-MM-DD').getMonth()` 在 UTC 负时区会出现月份标签边界问题。

## 17. 数据库与 Backup 影响

- 方案 A/B 可仅用现有活动 `notes`，不需要 Dexie 版本变更、迁移或 Backup 升级；但不能虚构时长或复盘。
- 方案 C 必须先定义长期活动事实的所有权、删除与恢复语义、索引、导入冲突和隐私边界。新表或字段意味着 Dexie 版本升级；若用户要跨设备恢复活动历史，则 Backup 需要扩展，按当前版本序列很可能成为 Backup v6。
- AIResult、KnowledgeAuditLog 目前已经被 Backup v5 覆盖，但不应为了热力图把它们重解释为学习活动。

## 18. 仍需运行时测量的问题

本审计未进行性能基准，也没有新增测试。以下问题必须在后续专项测量后再优化：

1. 500/2,000 篇笔记、3/5 年活动数据下 `loadAllNotes()`、日期聚合与筛选的耗时和内存。
2. Edge/8GB Windows 设备上的窄屏格子实际可点击尺寸、月份标签遮挡和原生 title 可发现性。
3. 年度与两年视图下 DOM、布局、筛选与日期详情抽屉的主线程成本。
4. 新学习会话数据的范围查询、聚合阈值稳定性、跨时区日键和 Backup 恢复成本。
