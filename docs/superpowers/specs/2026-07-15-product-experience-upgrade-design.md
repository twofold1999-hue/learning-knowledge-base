# 产品体验、课程学习工作台与性能优化设计

## 1. 文档定位与范围

本文档是当前有效的产品设计，用于归档下一阶段已确认的体验方向、边界和审计顺序。它不是实施完成报告，不是数据库设计，也不授权直接改动页面、Dexie schema、Backup 格式或 AI 流程。

产品下一阶段从“继续扩展 AI 和知识模型能力”转向“提升高频学习、编辑、复盘、可视化和长期使用体验”。后续实现必须先完成指定审计，再以小任务验证数据模型、交互和性能；不能依据本文直接进行一次性 UI 重写。

应用定位为本地优先的课程学习工作台，服务于计算机课程、绘画课程及其他系统学习场景。它不是单纯的 Markdown 编辑器，也不是知识管理后台。用户应能够：

1. 按课程、章节和课时持续学习；
2. 记录文字、代码、图片、练习、疑问与作品；
3. 将不同课程中的共同知识点建立关联；
4. 在学习后进行详细复盘或快速回顾；
5. 通过热力图和学习记录观察长期节奏；
6. 在中低配置 Windows 设备上长期稳定使用。

本设计保持本地优先：正文和知识数据以本地浏览器存储为事实来源，AI 只能由用户主动调用，备份与恢复必须在后续数据模型变更时同步评估。当前 Backup v5、现有 Dexie schema 和既有知识模型不因本设计而改变。

## 2. 已有能力与未来方向的边界

当前应用已经具备 Markdown 笔记、课程/学习单元、媒体学习、AI 整理与知识候选审核、AI 历史、知识实体关系、只读实体详情、笔记与实体双模式图谱、热力图、IndexedDB 本地存储和 Backup v5。

本文描述的课程工作台布局、混合编辑体验、学习计时、复盘记录、掌握度、热力图分层指标和性能诊断，除非明确标注为当前能力，否则均为后续方向。它们不得被实现或文档表述为已经存在。

课程结构、章节、课时、练习、作品、疑问和复盘的具体存储模型尚未在本设计中确定。后续专项设计必须先审计当前课程、项目、目录、笔记与学习单元的实际模型，再决定是否需要新表、字段、索引、迁移和 Backup 兼容策略。

## 3. 知识组织：课程树与跨课程知识图谱并存

目标组合结构为：

```text
课程
└─ 章节
   └─ 课时
      ├─ 学习笔记
      ├─ 练习或作品
      ├─ 疑问和错误
      └─ 本节复盘
```

课程树负责学习顺序、当前进度和课程语境；知识图谱负责跨课程、跨课时、跨笔记的概念关联。两者职责不同，不能相互替代。

- 一个知识点可以出现在多门课程、多个课时和多篇笔记中。
- 知识点不能永久锁定在单一课程树中。
- 现有 `KnowledgeEntity` 与 `KnowledgeRelation` 继续承担稳定、跨笔记的知识关联。
- 课程目录不能代替知识实体关系；知识图谱也不能代替课程顺序与学习进度。
- 后续如需给课程、章节或课时建立专用实体，必须先完成数据模型专项设计，不能在现有 Note 或 KnowledgeEntity 上临时拼接未定义字段。

## 4. 编辑器与学习工作台体验

### 4.1 默认页面结构

学习和编辑页面采用三栏能力、单一正文重心的工作台：

- 左侧显示课程、章节、课时目录、学习进度与快速切换入口。
- 中间是居中的主要学习和编辑区域，默认突出正文，不让辅助工具长期挤占阅读空间。
- 右侧默认完全收起；临时查看时使用覆盖式抽屉；用户可选择固定为推挤式面板。
- 固定状态与展开偏好可以持久化，但具体存储位置、范围和迁移策略属于后续实现设计。

右侧采用单一面板、多标签，不并列堆放多个永久展开的工具区。首批标签方向为：课程大纲、知识关联、复盘。每一标签都应按需读取和渲染；它们不能各自创建互不协调的全局状态或计时器。

### 4.2 正文与编辑模式

Markdown 继续是唯一正文数据源。增强能力通过 Markdown 语法、编辑器扩展和渲染组件实现，而不是再建立一套块 JSON、视觉编辑正文或双份正文保存。

目标编辑体验是混合式 Markdown：

- 默认呈现连续、适合阅读的编辑体验；
- 当前段落进入编辑时显示必要 Markdown 语法；
- 始终保留完整源码模式；
- 不牺牲 Markdown 的可移植性、可导出性和故障恢复能力。

此处是产品契约，不预设现有编辑器是否已经具备所需能力。编辑器审计应先确认当前 CodeMirror 配置、图片插入、预览、快捷键、长文档性能和可访问性，再确定最小实现路径。

### 4.3 内容宽度与专注

正文采用自适应混合宽度：普通文字正文的舒适阅读宽度约为 800px；代码块、表格、大图、绘画参考图和作品对比可以突破该宽度。提供专注模式与宽屏模式，且宽屏模式只改变展示布局，不创建第二份正文或改变笔记语义。

### 4.4 上下文工具栏

工具栏应按上下文出现，避免长期占据正文空间：

- 默认仅显示标题、保存状态、模式切换等核心操作；
- 选中文字时显示文本格式操作；
- 进入代码块时显示语言、复制、折叠等操作；
- 选中图片时显示说明、尺寸和对齐操作。

后续设计需先验证这些能力是否应由 CodeMirror 扩展、外层交互组件或 Markdown 渲染层承担；不得为了视觉效果重复保存格式化状态。

### 4.5 自动保存产品契约

目标自动保存行为：

1. 输入立即更新内存中的本地草稿；
2. 停止输入约 700～1000ms 后防抖写入 Dexie；
3. 界面明确显示“保存中 / 已保存 / 保存失败”；
4. 保存失败允许重试；
5. 切换笔记、离开页面和进入复盘前执行最终保存；
6. 不在每次按键时写 Dexie；
7. 自动保存不得阻塞输入。

该契约不代表当前实现需要重写。Task 20-A0 必须先测量当前保存频率、最终 flush、失败表现和输入流畅度，之后才定义任何改动。

## 5. 复盘与 AI 的边界

AI 功能在本阶段暂停扩张。不优先开发新的自动知识生成流程、复杂 AI Agent、后台自动分析或 AI 自动修改知识状态。

唯一保留的后续方向是“用户主动触发的快速回顾助手”。用户可选择当前课时、当前章节、整门课程或最近学习内容作为范围；AI 只输出临时的核心摘要、已记录的疑问和错误、建议重看的内容、原始笔记入口以及“开始自测”入口。

自测采用渐进辅助：先主动回答；困难时显示提示；仍困难时显示选项；最后显示解释和原始内容来源。AI 只负责临时压缩、提示和出题，禁止自动修改原始笔记、知识图谱或掌握度，也不得把 AI 总结冒充原始知识。

快速回顾不是 Task 20 第一批实施内容。现有 AIResult、AI 历史和知识审计仍各自承担当前职责：AIResult 记录生成及生命周期；KnowledgeAuditLog 记录知识结构变更；AI 历史只读展示不取代审计。

## 6. 复盘记录与掌握度方向

未来自测结果采用轻量事实记录，包含完成时间、复盘范围、总题数和正确率、错题、使用过的提示、薄弱知识点与完成状态。不长期保存完整 AI 对话。

掌握度使用三级：薄弱、熟悉、已掌握。系统可以根据自测提出建议，但必须由用户确认后更新。真实掌握度位于知识点层；课时层只汇总相关知识点的状态，不维护第二套彼此可能冲突的掌握度。

复盘记录、掌握度、错题、范围引用和可选 AI 题目来源都需要专门的数据模型、删除策略、隐私边界、Backup 兼容和迁移评估。它们不是当前 schema 的隐含字段。

## 7. 热力图体验设计

### 7.1 指标与强度

默认指标为综合学习强度，允许切换学习时长、练习完成、复盘次数和旧知识复习。每项指标独立归一化；分钟数与次数不得未经处理直接相加。

综合强度允许依课程类型使用不同默认侧重：计算机课程更重练习和代码实践；绘画课程更重练习时长和作品完成；理论课程更重学习时长、复盘和自测。权重只影响综合强度颜色，不修改原始活动数据。

采用个人基线：初始可参考最近约 90 天有效数据，阈值应稳定而非每日剧烈变化；两年对比使用同一阈值；悬停和详情始终展示真实分钟数及次数。

### 7.2 时间层级、筛选与视觉

时间层级为全年总览、月度放大、日期详情：

- 全年使用精致紧凑方格，默认只渲染一个年份，可切换年份，可选两年对比。
- 月度使用更大的日历卡片，展示日期、强度和简要课程活动。
- 日期悬停显示摘要；点击打开右侧抽屉；可继续进入完整日期复盘页。

默认显示全部学习活动，支持单门课程、多门课程组合、课程类型和全部课程。切换年份、月份和指标时尽量保留筛选状态。

视觉上以中性底色和单一强度渐变表示 0～4 级强度；课程颜色只用于标签、圆点、图例和日期详情标识，不能同时承担热力图强度含义。

这些均是目标体验。Task 20-B0 必须先确认现有热力图使用的数据来源、统计口径、日期边界和渲染成本，不能为了新版视觉直接虚构学习数据。

## 8. 学习计时器：后续数据基础设施

学习计时是热力图和复盘数据的后续独立子项目，不是本阶段直接实现内容。

### 8.1 计时与归属

采用混合计时：

- 普通自动计时记录编辑或阅读时的活跃时间；页面失焦后保留短暂宽限，超过配置时间且无活动时暂停。
- 手动专注学习可在切换到视频、IDE、文档或绘画软件后持续计时；用户主动暂停或结束；长时间无交互时提醒，而不是静默篡改记录。
- 自动暂停可选 3、5、10、15 分钟，默认 5 分钟。
- 学习会话必须归属课程，可选关联章节、课时、笔记和知识点。

一次会话只有一个当前主要活动类型，可有多个次要标签。切换主要活动类型不结束整次会话，而是结束当前时间片段并创建下一片段；片段分别统计，总时间不得重复累计。

### 8.2 编辑、恢复与浮动控制

结束会话后提供可视化时间轴和精确表单，允许修正时间片段边界、活动类型、开始结束时间、标签和归属。

学习过程中低频保存轻量检查点。浏览器异常关闭、系统崩溃或电脑休眠后，重新打开时由用户选择保留到最后活动时间、手动补齐或丢弃；休眠期间不能静默累计为学习时间。

全局悬浮迷你计时器仅在开始学习后渲染，默认右下角，可拖动、靠边吸附并记住停靠位置。展开后显示当前课程、主要活动类型、已用时间、暂停/继续、切换活动类型和结束学习。

性能边界明确：全应用只有一个计时状态；页面不能各自启动计时器；界面可按秒刷新，但不得每秒写 Dexie；只在检查点、切换片段、暂停、结束和 `pagehide` 等节点持久化；结束后清理定时器、监听器和临时状态。

## 9. 性能与内存目标

最低目标设备为 8GB 内存、较旧 CPU、集成显卡、Windows 和 Microsoft Edge。治理优先级依次为编辑输入流畅度、长时间使用后的内存稳定性、首次加载和页面切换速度。

性能原则：

- 先测量，后优化；不因猜测盲目添加 React.memo 或 useMemo。
- 编辑输入不能被统计、保存或知识查询阻塞。
- 图谱、热力图、AI 和复盘等重模块按需加载。
- 离开页面后清理大型对象、监听器和定时器。
- 大型列表仅在测量证明必要时引入虚拟化。
- 默认只渲染单年热力图。
- 实体图谱保持既有节点上限与降级策略。
- 动画轻量并尊重 `prefers-reduced-motion`。

性能规模基线：

| 规模 | 数据量 | 目标 |
| --- | --- | --- |
| 日常 | 100 篇笔记、20 门课程、1 年学习记录、数百实体与关系 | 高频功能流畅，无明显等待。 |
| 中等 | 500 篇笔记、50 门课程、3 年学习记录、数千实体与关系 | 编辑、课程浏览和热力图保持流畅；图谱允许明确、短暂加载。 |
| 压力 | 2000 篇笔记、100 门课程、5 年学习记录、上万实体与关系 | 可使用分页、节点上限、按需加载和功能降级，但不能崩溃、冻结或持续泄漏内存。 |

## 10. 性能诊断设计

设置页未来增加用户主动打开的“性能诊断”入口。日常模式不持续采集详细数据；只有用户主动打开或运行固定性能场景时才记录。

诊断关注编辑器首次可输入时间、输入响应、自动保存耗时、页面切换、热力图加载与筛选、图谱反复进入/退出、长任务数量、浏览器支持时的 JS Heap 趋势，以及离开页面后的内存回落。

验收结合人工体验、固定性能场景和关键指标记录；暂不建立容易受设备负载影响的严格提交门禁。诊断只保存最近 10 次摘要，支持导出 JSON，超过上限删除最早摘要；不保存逐秒原始数据。

导出默认脱敏，可包含对象数量、正文长度、内容类型、匿名 ID、页面类型、耗时、长任务、内存趋势、浏览器和设备基础信息。禁止包含笔记标题和正文、课程名称、图片内容和文件名、AI 提示词和输出、搜索关键词或用户输入文本。

## 11. 阶段顺序与可删除边界

Task 20 不进行一次性整体重构。下一步是三个独立、只读审计，可在隔离 Worktree 中并行：

1. Task 20-A0：编辑器体验、输入与保存链路审计。
2. Task 20-B0：热力图现有组件、数据来源与扩展成本审计。
3. Task 20-D0：性能、内存、包体和生命周期基线审计。

三个审计完成后执行 Task 20-01，整合审计报告、形成依赖图和实施路线。预计实现优先级为：编辑器输入与保存体验、编辑器布局与视觉、热力图数据契约与年度视图、热力图月度与日期详情、学习计时基础设施、性能与内存专项治理、可选轻量 AI 快速回顾。

新功能必须具备可删除边界：学习计时器、诊断、复盘和掌握度不应仅为 UI 便利写入核心笔记表；其入口、数据所有权、Backup 影响、定时器/监听器、测试和删除后残留必须在各自专项设计中说明。

## 12. 明确非目标

本阶段暂不：

- 扩展复杂 AI 功能、自动 AI 学习代理或 RAG；
- 让 AI 自动修改笔记、知识图谱或掌握度；
- 实现实体编辑、删除或合并；
- 将 Markdown 迁移为块 JSON；
- 全应用一次性视觉重写；
- 未经测量进行大规模性能重构；
- 为热力图直接虚构学习数据；
- 在未完成数据模型设计前修改 Dexie schema；
- 在未确定兼容策略前升级 Backup。

## 13. 已确认的产品决策

- 产品下一阶段聚焦高频学习、编辑、复盘、可视化和长期稳定性。
- 课程树与跨课程知识图谱并存，职责不互相替代。
- Markdown 保持唯一正文数据源。
- 学习工作台以中间正文为重心，右侧采用按需的单一多标签面板。
- 编辑器采用混合式 Markdown、上下文工具栏、专注与宽屏方向。
- 自动保存目标是内存草稿加防抖 Dexie 写入，并提供最终 flush 和可恢复失败状态。
- AI 暂停扩张；未来仅考虑用户主动触发的快速回顾助手。
- 复盘和掌握度以用户确认、知识点层事实为原则。
- 热力图使用独立归一化、个人基线、全年/月度/日期三级体验与课程筛选。
- 学习计时器是独立数据基础设施，拥有单一全局计时状态和低频持久化边界。
- 性能治理先测量后优化，以中低配置 Windows/Edge 长期稳定为目标。

## 14. 仍需审计或专项设计

- 当前编辑器是否支持美化 Markdown 编辑、上下文工具和宽屏布局。
- 当前自动保存的真实行为、写入频率、最终 flush、失败恢复与输入影响。
- 当前课程、章节、课时与学习单元模型是否足以承载课程树。
- 当前热力图的实际数据来源、统计口径、日期边界和扩展成本。
- 学习计时需要哪些新表、字段、索引和会话恢复策略。
- 掌握度、复盘记录、错题与范围引用的数据模型。
- 新学习数据进入 Backup 的兼容、恢复和版本策略。
- 当前首屏包体及大 chunk 的来源。
- 图谱卸载后的内存回落和 Edge 中可用的性能 API。
- 哪些体验可以先在不改 schema 的前提下实现。

本文不能把上述事项写成现有能力，也不能绕过 Task 20-A0、20-B0、20-D0 直接开始整体重写。
